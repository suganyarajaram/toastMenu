From the Integration Side

What I Found Building Against Toast's Menu API

These observations come from supervising an integration team working directly with Toast developer support on the Marketplace Facilitator approach. Every finding below is a real gap that affects operators today.

💸
Service charges injected as items inflate gross sales and break EOD reporting
Toast's external pricing approach does not support service charges. The only workaround is sending them as menu items. This inflates gross sales figures and means operators have to manually back-calculate their actual revenue from EOD reports.
Found during Marketplace Facilitator integration — confirmed by Toast developer support
How this should work: service charges as a first-class order entity, separate from menu items, excluded from gross sales calculation
CriticalMenus + Reporting
🏷️
No operator-facing visibility into marketplace price conflicts
When UberEats runs a promotional price on an item, it differs from the Toast menu price. Toast shows a balance due on the order. The operator has no dashboard to see which items have price conflicts across channels before orders start coming in.
Root cause of the UberEats Marketplace Facilitator requirement we had to implement
How this should work: a pricing conflict view showing in-store vs marketplace prices per item, with a one-tap resolution
CriticalMenus + Pricing
📡
Menu data unavailable for a location returns a 404 with no recovery path
When fetching menus for a location, a Channel not found error returns a 404. The operator or integration partner has no way to know whether the menu was never published, the location was misconfigured, or the channel simply does not exist. Dead end with no guidance.
Hit during live integration work across multiple restaurant locations
How this should work: error response explains why the channel was not found and what to do next
CriticalMenus API
🔄
Menu publish workflow is unclear for integration partners and operators
During integration work, the question of whether the standard publish workflow applied came up multiple times across multiple team members. The publish experience does not make it clear what gets published, to which channels, when, or whether downstream integrations will pick up the change automatically.
Observed across multiple back-and-forth messages with Toast developer support
How this should work: a pre-publish checklist showing exactly what will change, where, and for which channels
HighMenus UX
🧩
Zero-priced items with modifier pricing break discount logic
When a menu item has a base price of zero and its price comes entirely from modifiers, applying a discount throws an error. This is a menu data model edge case that affects any restaurant using complex modifier-based pricing — which includes a significant portion of fast casual and customisable menu restaurants.
Discovered and debugged during sandbox testing with Toast developer support
How this should work: discount logic that handles zero-priced items with modifier pricing as a supported pattern, not a workaround
HighMenus Data Model
🍽️
Multi-menu support not leveraged in third party integrations
Toast has strong multi-menu support. Most third party platforms do not. When integrating, operators have to manually consolidate their Toast menus into one because the integration layer does not expose the multi-menu structure. Toast's biggest menu differentiator gets flattened at the integration boundary.
Discovered while supervising Toast-Foodhub integration and building the multi-menu product
How this should work: integration APIs that expose multi-menu structure so third parties can map to it correctly
HighMenus API
🍕
The single insight behind all six findings
Toast's menu product is excellent from the inside. The gaps appear at the boundaries — where menus meet marketplaces, where prices differ across channels, where publish actions have unclear downstream consequences. The Menus PM who owns those boundaries is the most valuable PM on the team.

Four Versions Shipped

Menu API Evolution — V2 Through V4 and Beyond

Every version solved a real problem. Here is what each one added, why it was needed, and what it unlocked for operators and partners.

V2
Menu V2 — Internal
The Foundation — Internal Menu Management
Problem: No structured menu model. Agents manually managed menu data with no versioning, no consistency across locations.
Solution: A structured internal menu system with categories, subcategories, items, and modifiers. The foundation everything else is built on.
Categories Items Modifiers Internal use only
V3
Menu V3 — Partner API
Opening the Menu to the Outside World
Problem: Partners and third party integrators had no standard way to send their menu into our system. Every integration was custom, fragile, and unmaintainable.
Solution: A public-facing Partner API with a defined menu schema. Any company can now integrate and send their menu through a standardised contract I designed.
Public Partner API Standardised schema Tax handling Dietary labels Fulfillment modes
V3.5
Menu V3.5 — Selections
Supporting Complex Item Configurations
Problem: Partners needed to send items with nested selection structures — items where the choice of modifier itself defines the price and behaviour. V3 could not express this.
Solution: Selections support added to the menu schema. Items can now have complex nested modifier structures with min/max permitted, pre-selection defaults, and half-and-half support.
Selections Half N Half Pre-selected modifiers Nested modifier groups
V4
Menu V4 — Multi-Menu
One Restaurant, Many Menus — At Last
Problem: Restaurants needed different menus for different channels, times, and fulfillment modes. A single menu forced compromises. Breakfast vs dinner. Delivery vs dine-in. UberEats vs in-store.
Solution: Multi-menu support with service availability, time-based scheduling, fulfillment-specific menus, and channel-specific publishing. I benchmarked Toast's multi-menu architecture to design this.
Multiple menus Time-based scheduling Service availability Channel-specific publish Delivery vs In-store
+
Deals Extension — Grocery Integration
Extending the API for a Completely Different Menu Paradigm
Problem: A grocery company needed to integrate with us but their entire menu is structured as promotional deals — not restaurant items. The existing API had no concept of n-for-n, n-for-price-n, or meal bundles.
Solution: Extended the Partner API to support deal types as first-class menu entities. Flat discounts, percent discounts, buy-N-get-N-free, buy-N-for-price, and meal bundle structures — all with nutrition info, allergens, and time-based scheduling.
n-for-n promotions n-for-price-n Meal bundles Flat and percent discounts Nutrition info Allergens
Real PM Problem Solved

Menu ID Preservation — The Problem Nobody Saw Coming

When agents copy a menu from temp site to live, all the IDs change and the entire partner integration breaks silently. I identified this, defined the solution, got architecture sign-off, and broke it into 7 user stories.

❌ Before — What Was Happening
Agent edits menu on temp site
Menu copied back to live store
All item IDs change on copy
Partner integration references old IDs
Entire menu breaks for partner silently
Partner cannot process orders
→
✅ After — What I Designed
Store original IDs in partner_id column on export
On import: 3-stage DB transaction
① Update where ID exists in both
② Insert new items from payload
③ Delete items no longer in payload
Cross-store guard: different store → fallback to delete-recreate
Partner IDs preserved. Integration intact.
7 User Stories Written and Driven to Delivery
PA-11027
Preserve original menu IDs when copying from Temp to Live
PA-11028
Fall back to delete-and-insert when copying between different stores
PA-11029
Audit all menu flows doing full delete-and-insert vs partial update
PA-11030
Capture all edge cases for menu ID preservation change
PA-11031
Architect review and sign-off on menu ID preservation approach
PA-11032
Share Partner API menu upsert pattern as reference implementation
PA-11033
Validate and monitor partner sync after menu ID preservation ships
Why This Is the Most Important Thing on This Page

The JD says the Menus PM must ensure coherence across the menus product surface area. Menu ID preservation is exactly that — making sure a menu change in one part of the system does not silently break something else. I found this problem, designed the fix, wrote the spec including the 3-stage transaction pattern, got architect sign-off, and drove it to delivery. That is what a senior menus PM does.

Why This Changes Everything for Enterprise Customers

The JD specifically mentions customers managing menus across thousands of locations. Today if you publish a menu change and one location has a missing item or a broken channel, you only find out after the publish. For an enterprise with 500 locations, that is 500 potential failures you cannot see until something goes wrong. The Pre-Publish Confidence Score surfaces these before the publish, not after. That is the difference between a menu management tool and an enterprise-grade menu platform.
